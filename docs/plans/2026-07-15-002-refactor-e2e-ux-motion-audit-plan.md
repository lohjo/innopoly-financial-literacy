# E2E UX, motion, and polish audit (Mobbin-augmented)

## Overview

finfy-literacy has no shipped frontend yet (see `docs/plans/CURRENT_STATE.md` and the origin scope in `docs/plans/origin.md`). This plan is a **forward audit contract**: it defines, ahead of first implementation, the concrete UX/motion/polish bar every finfy-literacy surface must clear — onboarding, the challenge feed, the individual challenge/quiz screen, the post-challenge AI summary card, the leaderboard, streak/progress tracking, and profile — so those checks get built into the components from day one instead of retrofitted after a defect list accumulates.

Three lenses, used together:

1. **`make-interfaces-feel-better`** — 16 concrete design-engineering principles (concentric border radius, optical alignment, shadows-over-borders, interruptible animations, split-and-stagger enters, contextual icon animations, font smoothing, tabular numbers, text-wrap, image outlines, scale-on-press, skip animation on page load, never `transition: all`, sparing `will-change`, 40×40px hit area, subtle exits).
2. **`web-animation-design`** — motion timing, easing curves, transition specificity, GPU acceleration, `prefers-reduced-motion` compliance.
3. **Mobbin MCP** (`https://api.mobbin.com/mcp`), cross-referenced against the local Fingo reference mockups at `docs/audit/fingo-references-png/` (`fingo-001.png` … `fingo-006.png`) — real production-app patterns per surface family, used as benchmarks to prevent AI-slop reflexes once the build starts.

finfy-literacy is explicitly a **duolingo-style, flat-card, mobile-first** product (per `docs/plans/origin.md`), not a translucent-chrome-over-a-3D-scene product. Every principle below is applied through that lens: elevation and shadow carry depth, not background transparency; tap targets and active/press states matter more than hover states; and the leaderboard's whole premise — a **real peer group, not anonymous strangers** (per `docs/plans/features/leaderboards-peer-competition.md`) — means loading and empty states must never make peers look anonymous or absent by accident.

## Problem Frame

There is no shipped code to recon yet, so these are **anticipated defect classes**, not confirmed bugs — drawn from the product's own risk surface (a strict-format AI summary, a real-name leaderboard, streak-driven habit mechanics) and from the class of bugs that recur in gamified, animation-heavy mobile apps generally. Each should be treated as a checklist item during initial build, not discovered after the fact:

- **Onboarding**: the peer-group/leaderboard invite step risks a flash of an unstyled or "no peers yet" empty state before real peer data resolves — which, for a product whose whole differentiator is a *real* peer group, reads as broken trust rather than a loading state.
- **Post-challenge AI summary card**: the strict fixed format (what you did / score / one next step / track position, per `docs/plans/features/ai-generated-summaries.md`) is generated by an LLM call and risks rendering as a streaming or partially-parsed block before the full structured response is ready — the format's entire value proposition is that it reads the same way every time, so a half-rendered state undermines it.
- **Streak / progress tracking**: the streak-flame or progress-ring element risks rendering in its "lit" or "complete" end state for one frame before its reveal animation plays — the same class of render-before-animate flash that undermines any celebratory motion.
- **Leaderboard**: rank-change and points-update transitions risk racing each other (a row re-sorting via FLIP while its own points chip is mid-pulse), and slow-loading peer avatars risk flashing a placeholder that looks like an anonymous stranger rather than a named peer.
- **Profile**: the header/streak-summary band risks becoming a flat, disconnected block of color that doesn't share the elevation/shadow language of the rest of the app — the mobile-card equivalent of the "opaque chrome" failure mode.
- **Challenge feed**: locked (not-yet-unlocked) challenge cards and any status badges risk using ad hoc, mismatched opacity instead of one consistent scrim/elevation system across cards.
- **Invite / share overlay**: an "invite a friend" or "share your streak" sheet — launched from onboarding, the leaderboard, or profile — risks opening *behind* whichever screen invoked it, or losing its anchor point across open/close cycles.
- **Cross-flow**: motion timings, 40×40 tap targets, active/press feedback, tabular numbers, and font smoothing have no default audit pass built into the product yet — without one, these details compound into a surface that "feels generated" the same way they would in any freshly-built app.

The shared risk: no single principle failure is dramatic on its own, but many small unaudited details compound into a product that feels assembled rather than designed. The goal of this plan is to get ahead of that by writing the audit checklist before the components exist.

## Requirements Trace

- R1. The post-challenge celebration (points tally, streak-flame/ring reveal, AI-summary-card entrance) never renders its settled end state before its reveal animation begins.
- R2. The invite/share overlay always renders above whichever screen invoked it (onboarding, leaderboard, profile), with correct stacking context, and re-anchors correctly across open/close cycles and across different entry points.
- R3. Profile's header/streak-summary band shares the same elevation and shadow language as the rest of the app; it never presents as a flat, disconnected block of chrome.
- R4. Challenge feed, leaderboard, and AI-summary cards share one consistent scrim/elevation system; locked-challenge overlays, rank badges, and status pills never appear as mismatched flat-opaque patches on a card.
- R5. Every interactive element (challenge card, quiz answer option, leaderboard row, bottom-nav tab, streak/points chip, invite button) has a minimum 40×40px hit area and a `scale(0.96)` press feedback.
- R6. Every nested rounded element (thumbnail-in-card, avatar-in-leaderboard-row, badge-in-chip) uses concentric border radius (outer = inner + padding).
- R7. Every dynamic numeric display (points counter, streak-day count, leaderboard rank, quiz score, countdown timer) uses `font-variant-numeric: tabular-nums`.
- R8. The root layout applies `-webkit-font-smoothing: antialiased`.
- R9. Headings (challenge titles, AI-summary headline) use `text-wrap: balance`; body copy (challenge instructions, AI-summary next-step text, empty-state copy) uses `text-wrap: pretty`.
- R10. No `transition: all` rules anywhere in the codebase; every transition specifies exact properties.
- R11. `will-change` is only applied to `transform`/`opacity`/`filter`, only where first-frame stutter is actually observed.
- R12. All transitions ≥ 200ms collapse to ≤ 80ms when `prefers-reduced-motion: reduce` is set.
- R13. The full E2E flow (cold signup → onboarding/peer-invite → challenge feed → individual challenge/quiz → AI summary card → leaderboard → streak/progress → profile → sign out) is captured in before/after screenshots and verified end-to-end once the surfaces exist.

## Scope Boundaries

- **Not** redesigning information architecture. (No merging the challenge feed and leaderboard, no reshuffling the core loop from `docs/plans/finfy-literacy-phase-0-planning.md`.)
- **Not** reopening any MVP scope call from `docs/plans/origin.md` — financial planning tools/advice, institution integrations, advanced analytics, social messaging/forums, monetization, and gamification mechanics beyond points + streaks + leaderboards stay out of scope for this audit exactly as they're out of scope for the product.
- **Not** changing the AI summary's strict-format schema itself (fields, order, content) — that schema is owned by `docs/plans/features/ai-generated-summaries.md`. This plan only audits how the fixed-format card renders and animates, not what it says.
- **Not** picking a frontend framework or component library. This plan defines surface-level contracts (visibility gates, stacking rules, elevation system, tap targets, motion timing) that apply regardless of which mobile-first web stack the MVP build lands on.
- **Not** changing prose/copy unless a specific cognitive-load or AI-slop issue is identified during the audit.

### Deferred to Follow-Up Work

- Tablet/desktop responsive scaling beyond the mobile-first baseline — per `docs/plans/features/mobile-first-design.md`, the MVP targets a mobile-first responsive web app; larger-viewport polish and the native-app evaluation are later-phase decisions, not this plan's scope.
- A design-tokens layer extracted from per-screen styles, once real components exist to extract tokens from.
- Push-notification streak-risk motion/UX (banner entrance, badge animation) — deferred until push notifications ship, per the phase-tracking table in `docs/plans/features/mobile-first-design.md`.
- The two secondary future surfaces named in `docs/plans/origin.md` — a high-school-student track and an "AI video call" mascot feature — are explicitly out of MVP scope and are not audited here.

### Motion-suspect shortlist

Motion mechanisms most likely to develop timing bugs once built, based on how each is expected to be implemented:

- Points-counter count-up animation racing the AI-summary-card's own fade-in (two independent animation loops mounting together).
- Streak-flame "lit" state swap implemented as a `display: none → block` toggle instead of an opacity/scale transition.
- Leaderboard rank-change reorder (FLIP-style repositioning) racing a row's own points-update pulse.
- Onboarding → first-challenge cross-fade race (challenge feed mounts before the prior screen's exit transition finishes).
- Quiz-answer correct/incorrect feedback flash racing the "next question" auto-advance timer.
- Invite/share-sheet send-invite spinner with no stop condition if the invite call fails silently.
- Challenge-complete celebration overlay's portal-mount timing relative to the AI-summary-card's own entrance.

## Key Technical Decisions

- **Mobbin + Fingo first, opinion second.** For each surface family, pull 2–3 Mobbin reference patterns and cross-check against the matching local Fingo mockup before designing any component. The references live in `docs/audit/mobbin-references-2026-07-15.md`.
- **Foundational correctness before visual polish, built in from the start.** Visibility gating (R1), overlay stacking (R2), and the elevation system (R3, R4) are treated as component contracts from first implementation — not a defect list to clean up after the fact. Everything else in this plan (typography, surfaces, animation, performance, hit areas, coverage gaps) assumes those contracts are already honored.
- **The `make-interfaces-feel-better` review-output convention.** Every polish-sweep unit (U6 onward) presents findings as a before/after table grouped by principle. No prose-only findings.
- **Verification by replay.** Each unit's verification step recreates the affected user flow from a known state (a fresh account, or a cleared local session) once the corresponding surface exists to replay against.

- U2. **Define the pre-reveal visibility contract for celebration components**

**Goal:** The post-challenge celebration (points tally, streak-flame/ring reveal, AI-summary-card entrance) is never visible in its settled end state before its reveal animation begins.

**Requirements:** R1

**Dependencies:** None (foundational component contract)

**Files:**
- Planned: the challenge-completion / celebration component and whatever mounts it from the quiz screen.

**Approach:**
- Define a visibility-gate pattern at the component level: celebration elements start in an explicit pre-reveal state (opacity 0 / scale 0.25, or a "not yet lit" flame variant) set synchronously before first paint — never toggled visible after mount and then animated.
- The transition from quiz screen → AI-summary card sets starting values before the animation loop begins, so the first rendered frame is always the pre-reveal state, never the settled one.
- Reduced-motion path: reveal directly at the settled state (no count-up/arc), matching the reduced-motion carve-out used everywhere else in this plan.

**Test scenarios:**
- *Happy path:* Complete a challenge → AI-summary card and celebration elements appear in pre-reveal state, then animate to settled. No pre-reveal flash of the settled state.
- *Edge case:* Refresh mid-transition (e.g., between quiz submission and summary render) — resumes at the correct visual state.
- *Reduced motion:* `prefers-reduced-motion: reduce` — celebration reveals instantly at the settled state, never visible-before-revealed.

**Verification:**
- Screen recording of a challenge completion shows no pre-reveal flash frame, once the component exists.

- U9. **`make-interfaces-feel-better` pass — animations**

**Goal:** Apply the animation principles (interruptible transitions, split-and-stagger enters, subtle exits, contextual icon animations, scale-on-press, skip-on-load).

**Requirements:** R10, R11 (R1 stays satisfied)

**Dependencies:** U2 (visibility-gate contract), U7+U8 (visual is in place first)

**Files:**
- Planned: shared transition styles; the screen/card-mount transition component; icon-context components (correct/incorrect answer icon, streak-flame lit/unlit, rank-change arrow).

**Approach:**
- **No `transition: all`:** once styles exist, grep for it and replace with specific properties.
- **Scale on press:** every card/button/tab/chip/row — `&:active { transform: scale(0.96); }`, `transition-property: transform, opacity`, `transition-duration: 120ms`. Never below 0.95.
- **Split-and-stagger enters:** AI-summary-card fields enter in sequence (headline → score → next step → track position), each ~200ms with a staggered ~80ms delay per field; challenge feed and leaderboard rows stagger similarly on first load.
- **Subtle exits:** screen/card exits are a plain fade, softer than the enter — no scale or translate.
- **Contextual icon animations:** correct/incorrect answer icon swap, streak-flame lit/unlit swap, rank-up/rank-down arrow — `opacity 0→1, scale 0.25→1, blur 4px→0px`, `cubic-bezier(0.2, 0, 0, 1)`.
- **`prefers-reduced-motion`:** sweep every transition above ≥ 200ms and add an override collapsing it to 80ms.

**Test scenarios:**
- *Card enter:* AI-summary card fields fade in staggered — headline first, then score, then next step, then track position.
- *Card exit:* Dismissing the summary card is an instant fade, no stagger.
- *Press feedback:* Tapping any quiz option, card, or tab shows a quick 0.96 scale.
- *Reduced motion:* OS setting on → enters/exits collapse to 80ms, staggers compress.
- *No more `transition: all`:* grep returns zero matches.

**Verification:**
- Short screen recording of the AI-summary-card reveal showing the stagger, once built; before/after table per principle.

## System-Wide Impact

- **Interaction graph:** the invite/share overlay lives inside whichever screen's stacking context invoked it — when that screen changes, the overlay should be collected too. The eventual implementation must not assume the overlay survives a screen change it wasn't opened from.
- **Error propagation:** new error banners (challenge-submit failure, AI-summary-generation failure) must clear on success and never stack on top of each other.
- **State lifecycle risks:** overlay re-portaling on each open (U3) is the safety property to preserve — no code path should open the invite/share overlay without re-portaling first.
- **API surface parity:** none changed by this plan. The AI summary's strict-format schema and the leaderboard's real-peer-only model are explicitly out of scope (see Scope Boundaries) and must remain untouched by any unit here.
- **Integration coverage:** the Mobbin MCP integration is a new external dependency for this audit — the brief lives on disk (`docs/audit/mobbin-references-2026-07-15.md`) so the audit doesn't repeatedly re-fetch.
- **Unchanged invariants:** the AI-generated-summary strict format (`docs/plans/features/ai-generated-summaries.md`), the leaderboard's real-peer-group model (`docs/plans/features/leaderboards-peer-competition.md`), and the MVP scope boundary in `docs/plans/origin.md` (no badges/achievements beyond points + streaks + leaderboards).

## Risks & Dependencies

| Risk | Mitigation |
|------|------------|
| Mobbin MCP not connected at execution time | Confirm tool availability before starting U1; if unavailable, fall back to `make-interfaces-feel-better` principles and the local Fingo mockups alone |
| The existing `docs/audit/mobbin-references-2026-07-15.md` still describes a prior, unrelated product | U1 fully rewrites the file rather than appending to it; do not cite its current on-disk content as valid reference material until rewritten |
| Profile header elevation treatment reads as washed-out or too subtle once real content is in it | Visual check during U4 against real streak/points data, not placeholder text; tune shadow/radius values if needed |
| Split-and-stagger AI-summary-card reveal feels slower than a flat fade (≈360ms total vs. ≈200ms) | If it reads as sluggish once built, compress to a 60ms per-field stagger or accept the slower reveal as intentional pacing |
| Hit-area extension via pseudo-elements creates click-through-elsewhere bugs on dense screens (leaderboard rows, quiz options) | The skill explicitly bans overlapping hit areas — keep any extension to ≤ 4px per side |
| `tabular-nums` on points/streak counters changes visual width unexpectedly next to icons | Spot-check once built; if too monospace-heavy, apply only where the value actually updates dynamically |
| `transition: all` sweep changes press/active behavior somewhere unintended (mobile has no hover to fall back on) | Verify each rewrite preserves the original animated property set, tested via tap not hover |
| This plan is written before any frontend code exists | Revisit each unit's "Files" section at build kickoff and replace planned descriptions with real paths |
