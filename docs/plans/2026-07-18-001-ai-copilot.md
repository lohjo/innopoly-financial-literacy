## What Koji actually is, and what "minimal" means here

From the blog post [^1]: it (1) sees the exact lesson state — the interactive you're stuck on, your wrong guesses, where you paused; (2) reaches *into* the lesson to highlight/annotate/pose a follow-up question rather than just replying in a text box; (3) speaks first, unprompted, at the start of each page — noticing what changed, not reading instructions; (4) never gives the answer, it asks; (5) has a character/face because a generic sparkle button tested worse. ## The two things that make Koji feel like Koji, not a chat widget

1. **Screen/state awareness, not screen recording.** Koji doesn't watch pixels — it has structured access to lesson state (current step, learner's last 3 attempts, which interactive element is active). You should do the same: don't feed Gemini Live raw video of the canvas (expensive, imprecise); feed it a structured JSON snapshot of lesson state on every state change, and only use actual screen/video capture if you truly need OCR-style visual grounding (e.g. reading a hand-drawn graph). For your Brilliant-clone lessons, structured state is strictly better and far cheaper.
2. **It acts, not just talks.** The killer feature is Koji reaching into the lesson — highlighting a region, annotating a graph. Minimal version: define 3–5 tool calls the model can invoke (`highlight_element(id)`, `show_hint(level)`, `pose_followup(question_id)`, `annotate_graph(points)`) and have the client execute them against the live lesson DOM. Without this, you've built a voice chatbot next to a lesson, not a tutor inside it — and that's the difference the whole blog post is about.
